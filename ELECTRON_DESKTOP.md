# Electron Desktop Packaging

This document is the operating guide for the Windows Electron desktop client. It covers local development, installer builds, installer-level updates, GitHub Releases, ModelScope fallback assets, and runtime data locations.

## Overview

The desktop client is made of four parts:

1. The Electron shell in `electron/`.
2. The FastAPI backend from `main.py`.
3. A PyInstaller backend bundle written to `dist/infinite-canvas-backend/`.
4. An electron-builder NSIS installer written to `release/`.

The update system depends on the three files generated by electron-builder:

```text
Infinite-Canvas-Setup-<VERSION>.exe
Infinite-Canvas-Setup-<VERSION>.exe.blockmap
latest.yml
```

The same three files must be published to both GitHub Releases and the ModelScope fallback directory. The filenames referenced by `latest.yml` must match the uploaded files exactly.

## Development

Run the Electron development client with:

```bat
启动Electron开发版.bat
```

The Electron main process starts the local FastAPI backend and opens a desktop window.

Electron installer updates are disabled in development mode. The update flow only runs when `app.isPackaged` is true.

## Build Installer

Build the Windows installer with:

```bat
npm run build:win
```

The Windows helper script runs the same build flow:

```bat
打包Electron桌面版.bat
```

The build flow is:

1. `npm run sync:desktop-version` reads `VERSION` and updates Electron package metadata.
2. `npm run build:backend` packages the FastAPI backend with the project `venv`.
3. `electron-builder --win nsis` packages the Electron shell and embeds the backend bundle.
4. The NSIS installer, `.blockmap`, `latest.yml`, and `win-unpacked/` output are written to `release/`.

For a directory-only package, use:

```bat
npm run pack:win
```

Build outputs are intentionally ignored by Git:

```text
dist/
release/
build/backend/
```

Do not expect `git push` to publish installers. Installers are distributed through GitHub Releases and ModelScope assets.

## Versioning

The visible installer suffix comes from the first line of the root `VERSION` file. For example, if `VERSION` contains:

```text
2026.07.6
```

the installer should be generated as:

```text
release/Infinite-Canvas-Setup-2026.07.6.exe
```

`npm run sync:desktop-version` also writes a semver-compatible value to `package.json.version` and `package-lock.json`. That metadata may omit leading zeros, so `2026.07.6` becomes `2026.7.6` for Electron metadata while the installer filename keeps the original `VERSION` text.

Project releases should use a bare numeric three-part `VERSION`: `MAJOR.MINOR.PATCH`. Do not add a leading `v` in `VERSION`; release commands add the tag prefix separately as `v<VERSION>`, and the installer filename uses the raw `VERSION` text. Do not use a four-part value such as `2026.07.6.1` unless the version sync script is changed first.

Check the expected installer name without running a full build:

```bat
npm run sync:desktop-version
```

Do not manually rename the generated installer. `latest.yml`, the `.blockmap` path, and update downloads all depend on the generated filename.

## Backend Packaging Environment

The packaged backend must be built with the project virtual environment, not a global `pyinstaller` from `PATH`.

The backend build script is:

```bat
node scripts/build-backend.cjs
```

It does the following:

1. Creates `venv` if `venv\Scripts\python.exe` is missing.
2. Installs missing `requirements.txt` runtime dependencies into `venv`.
3. Installs PyInstaller into `venv` if needed.
4. Verifies `httpx` can be imported.
5. Runs `venv\Scripts\python.exe -m PyInstaller` with `build/backend.spec`.

This protects packaged clients from starting with missing backend dependencies such as `ModuleNotFoundError: No module named 'httpx'`.

## Client Update Model

### Installer-Level Updates

Packaged Electron clients use `electron-updater` for installer-level updates. When an update is available, the client asks the user before downloading. After download, it asks again before restarting and installing.

After the user chooses `下载更新`, Electron opens a small main-process progress window and updates the Windows taskbar progress indicator. The progress window shows the active source, percentage, transferred size, total size when available, and download speed. Closing that progress window does not cancel the installer download; the download continues in the background and the window can be restored by clicking the sidebar `检查客户端更新` entry while the state is still downloading.

Before installation, Electron stops the local backend process and then calls `autoUpdater.quitAndInstall()`.

### GitHub Primary Source

GitHub Releases are the primary update source:

```text
owner: xiaoqi0102
repo: Infinite-Canvas
releaseType: release
```

`package.json` keeps `build.publish` configured for GitHub only. Do not turn this into a dual publish configuration. ModelScope is a runtime fallback provider implemented in `electron/main.js`, not an electron-builder publish provider.

### ModelScope Fallback Source

If the GitHub update check or download fails, the client switches to ModelScope and retries the same installer-level update flow.

Default ModelScope target:

```text
API root: https://modelscope.cn/api/v1/studio
owner: xiaoqi0102
repo: Infinite-Canvas
revision: master
directory: desktop-release
```

The fallback file API path is:

```text
https://modelscope.cn/api/v1/studio/xiaoqi0102/Infinite-Canvas/repo?Revision=master&FilePath=desktop-release/<file>
```

The fallback source can be redirected without code changes:

```text
INFINITE_CANVAS_CLIENT_UPDATE_MODELSCOPE_API_ROOT=https://modelscope.cn/api/v1/studio
INFINITE_CANVAS_CLIENT_UPDATE_MODELSCOPE_OWNER=xiaoqi0102
INFINITE_CANVAS_CLIENT_UPDATE_MODELSCOPE_REPO=Infinite-Canvas
INFINITE_CANVAS_CLIENT_UPDATE_MODELSCOPE_REVISION=master
INFINITE_CANVAS_CLIENT_UPDATE_MODELSCOPE_DIR=desktop-release
```

### Manual Check Entry

Manual client update checks are available in the desktop web UI sidebar. The button text is `检查客户端更新`, and it sits below the version badge near the lower-left author area.

The button calls a narrow Electron preload IPC bridge and triggers the same installer-level update flow as the delayed background check.

### Difference From In-Page Source Update

The web page "one-click update" flow is not the Electron installer update flow.

The in-page update reminder still targets source-project files such as `main.py`, `VERSION`, and `static/`. Do not repurpose it for installer-level updates.

### Diagnostics

Client update diagnostics are appended to `InfiniteCanvas_Data\desktop.log`.

Important update events include:

```text
client-update-source-selected
client-update-checking-for-update
client-update-source-fallback
client-update-update-available
client-update-download-progress
client-update-progress-window-closed
client-update-duplicate-download-error-ignored
client-update-update-downloaded
client-update-update-error
```

Logs include the active source, either `github` or `modelscope`, whenever source-specific behavior matters. Download progress is logged periodically rather than on every raw progress event, while the visible progress window is refreshed more frequently.

See `Runtime Data` for how `InfiniteCanvas_Data` is selected and where `desktop.log` is written.

## Release Checklist

### 1. Preflight

Confirm the workspace is clean and the version is the intended release version:

```powershell
git status --short --branch
$v = (Get-Content -LiteralPath VERSION -TotalCount 1).Trim()
$v
```

Run lightweight syntax checks before building:

```powershell
node --check static\js\smart-canvas.js
node --check electron\main.js
node --check scripts\build-backend.cjs
node --check scripts\sync-electron-version.cjs
```

### 2. Build

Build the Windows installer:

```powershell
npm run build:win
```

### 3. Validate Local Artifacts

Check that all three update artifacts exist:

```powershell
$v = (Get-Content -LiteralPath VERSION -TotalCount 1).Trim()
Test-Path "release\Infinite-Canvas-Setup-$v.exe"
Test-Path "release\Infinite-Canvas-Setup-$v.exe.blockmap"
Test-Path "release\latest.yml"
Get-Content "release\latest.yml"
```

`latest.yml` must reference `Infinite-Canvas-Setup-$v.exe`, and its `size` must match the generated installer.

### 4. Publish GitHub Release

Create a new GitHub Release with the same three artifacts:

```powershell
$v = (Get-Content -LiteralPath VERSION -TotalCount 1).Trim()
gh release create "v$v" `
  "release\Infinite-Canvas-Setup-$v.exe" `
  "release\Infinite-Canvas-Setup-$v.exe.blockmap" `
  "release\latest.yml" `
  --repo xiaoqi0102/Infinite-Canvas `
  --title "Infinite Canvas Desktop v$v" `
  --latest
```

If you intentionally replace assets on an existing Release, use `gh release upload ... --clobber` and verify the remote asset hashes afterward.

### 5. Upload ModelScope Fallback Artifacts

Upload the same three files to the ModelScope Studio repo under `desktop-release/`.

Use this order:

1. Installer `.exe`
2. Installer `.exe.blockmap`
3. `latest.yml`

Uploading `latest.yml` last prevents old clients from seeing new metadata before the referenced files are available.

Prefer the Python Hub API for the Studio repo. The `modelscope upload` CLI may only support `model` and `dataset` repos in some installations.

```powershell
@'
from pathlib import Path
from modelscope.hub.api import HubApi

repo = 'xiaoqi0102/Infinite-Canvas'
revision = 'master'
version = Path('VERSION').read_text(encoding='utf-8').splitlines()[0].strip()
files = [
    (Path('release') / f'Infinite-Canvas-Setup-{version}.exe', f'desktop-release/Infinite-Canvas-Setup-{version}.exe'),
    (Path('release') / f'Infinite-Canvas-Setup-{version}.exe.blockmap', f'desktop-release/Infinite-Canvas-Setup-{version}.exe.blockmap'),
    (Path('release') / 'latest.yml', 'desktop-release/latest.yml'),
]

api = HubApi()
for local, remote in files:
    api.upload_file(
        repo_id=repo,
        repo_type='studio',
        path_or_fileobj=str(local),
        path_in_repo=remote,
        revision=revision,
    )
'@ | .\venv\Scripts\python.exe -
```

Install `modelscope` into the project `venv` first if needed:

```powershell
.\venv\Scripts\python.exe -m pip install modelscope
```

### 6. Validate Remote Artifacts

For GitHub, inspect the Release assets:

```powershell
$v = (Get-Content -LiteralPath VERSION -TotalCount 1).Trim()
gh release view "v$v" --repo xiaoqi0102/Infinite-Canvas --json tagName,name,url,assets
```

For ModelScope, validate with `GET` requests against the file API. Do not use `HEAD`; the ModelScope file API may return `404` for `HEAD` even when `GET` works.

Use a streaming `GET` for the large installer so verification does not download the full file:

```powershell
@'
from pathlib import Path
from urllib.parse import urlencode
import requests

version = Path('VERSION').read_text(encoding='utf-8').splitlines()[0].strip()
base = 'https://modelscope.cn/api/v1/studio/xiaoqi0102/Infinite-Canvas/repo'
files = [
    f'desktop-release/Infinite-Canvas-Setup-{version}.exe',
    f'desktop-release/Infinite-Canvas-Setup-{version}.exe.blockmap',
    'desktop-release/latest.yml',
]

for file_path in files:
    url = base + '?' + urlencode({'Revision': 'master', 'FilePath': file_path})
    with requests.get(url, stream=True, timeout=120) as response:
        print(file_path)
        print('  status:', response.status_code)
        print('  content-length:', response.headers.get('content-length', ''))
        response.raise_for_status()
        if file_path.endswith('latest.yml'):
            text = response.text
            print('  references installer:', f'Infinite-Canvas-Setup-{version}.exe' in text)
        else:
            first_chunk = next(response.iter_content(chunk_size=64), b'')
            print('  first bytes:', len(first_chunk))
'@ | .\venv\Scripts\python.exe -
```

### 7. Smoke Test Installed Client

After publishing, install the generated package on a test machine or clean test directory.

Check that the desktop client starts, the backend launches, `InfiniteCanvas_Data\desktop.log` is written, the `检查客户端更新` button is visible, and user data is created outside the installation directory.

Also smoke test the smart canvas loop node after UI changes: create a loop node with many prompt variants, confirm the prompt list scrolls inside the node, resize the node manually, and verify the footer controls stay visible and clickable.

## Runtime Data

The desktop app stores writable user data in an `InfiniteCanvas_Data` folder next to the installation directory, not inside it. For example, if the app is installed at `D:\Apps\Infinite Canvas`, user data is stored at `D:\Apps\InfiniteCanvas_Data`.

Electron passes this path to the backend through `INFINITE_CANVAS_USER_DATA_DIR`. The legacy `INFINITE_CANVAS_BASE_DIR` variable is still passed with the same value for compatibility.

If the parent directory of the installation directory cannot create or write to the sibling data folder, Electron falls back to an `InfiniteCanvas_Data` folder under Electron's system user data directory.

On startup, Electron copies missing files from the old in-install `InfiniteCanvas_Data` folder into the new sibling folder when that old folder exists. The backend keeps static files and built-in workflows read-only from the packaged app. User data such as API settings, canvases, uploads, custom workflows, history, and outputs are written under `InfiniteCanvas_Data`.

The desktop launcher appends startup diagnostics to `desktop.log` inside the active `InfiniteCanvas_Data` folder. The log records the selected data directory, install/resource paths, backend command, port, backend exit code, and startup errors.

The NSIS installer sets `deleteAppDataOnUninstall` to `false`, so uninstalling the desktop app does not delete Electron's system app data. The primary user data is outside the install directory in the sibling `InfiniteCanvas_Data` folder and should also be preserved across reinstalls and updates.

## Maintenance Notes

Keep these points intact when changing the desktop packaging or update flow:

1. GitHub and ModelScope must receive the same three artifacts: installer, `.blockmap`, and `latest.yml`.
2. `latest.yml` must reference the exact installer filename that exists on both sources.
3. ModelScope upload order is installer first, `.blockmap` second, `latest.yml` last.
4. ModelScope remote validation should use `GET`, not `HEAD`.
5. `package.json` `build.publish` should stay GitHub-only.
6. ModelScope should remain a runtime fallback provider in `electron/main.js`.
7. `formatVersionLike()` and `versionTokenFromPath()` protect `.blockmap` paths when `package.json.version` omits leading zeros but installer filenames keep them.
8. Electron installer updates only run in packaged clients, not development mode.
9. The web page "one-click update" is still for source-project updates, not installer-level updates.
10. The visible installer download progress belongs to the Electron main-process update flow. Do not move it into the in-page source update flow or require broad renderer IPC exposure.
11. User data must stay outside the installation directory and survive uninstall/reinstall cycles.
